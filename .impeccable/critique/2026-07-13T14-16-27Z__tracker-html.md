## Anti-Patterns Verdict

**LLM assessment:** Passes every hard ban (no side-stripes, no gradient text, no glassmorphism, no modal-first — though native confirm()/alert() are close cousins). Fails the register floor: the top-grid composition (donut + ring + four identical KPI cards + progress bars) is the interchangeable admin-template look PRODUCT.md names as its own anti-reference. Two same-size 200px circles read as chart gallery, not mission control. Emoji in a section header is a template tell. The authored, personal elements (commitment grid, honest chart coloring) are buried mid-page.

**Deterministic scan:** 1 finding — `layout-transition` at tracker.html:31 (`transition: width` on `.fill` progress bars), confirmed true positive that the LLM review missed. Low practical severity, and the fasting ring already uses the compositor-friendly `stroke-dashoffset` pattern; the bars should match. No false positives.

**Visual overlays:** skipped — no browser automation available this session; evidence base is static scan + source verification.

## Overall Impression

Solid engine, template face, wrong order. The data modeling (fast slicing, honest deltas, backdate guards) is better than the presentation, and the page is laid out for desktop review while the product's life happens on a phone at 6 AM. Biggest single opportunity: reorder mobile around the daily loop and give the page its missing voice.

## What's Working

1. **The 6-month commitment grid** — 184 days as one shape, missed days plainly gray. Makes the invisible middle visible; it's the one element no template has.
2. **Honest color semantics in the data** — gains render red with a "+", every up-tick is visible. Principle 5 executed at detail level.
3. **Fasting domain modeling** — midnight-slicing for 16h+ days, streak tolerant of today-in-progress, specific guard copy. Thoughtful engineering.

## Priority Issues

1. **[P1] Mobile stacking order inverts "log in seconds."** At 640px: nav → header → settings → donut → fasting card → KPIs → bars → weigh-in (~2 screens) → check-ins (~3 screens). The daily actions are the last things reached. Fix: mobile order = fast card, weigh-in, check-ins, then stats; collapse sign-out/log-date/plan-start behind one settings row. → /impeccable adapt
2. **[P1] First paint shows wrong data.** Auth gate flashes, then hardcoded 258.4/"0 days"/"Day 1 of 184" until data lands. The app greets him with his heaviest number every morning. Fix: neutral placeholders + minimal loading treatment. → /impeccable harden
3. **[P1] Daily-loop ergonomics miss the stated floors.** Primary buttons ~38px, pills ~33px (44px floor); 0.9rem inputs trigger iOS focus-zoom; Enter does nothing; no optimistic state on habit taps; errors surface off-screen. → /impeccable adapt + harden
4. **[P2] The commitment grid contradicts his actual program.** A day counts only if BOTH workout AND meal are true (verified tracker-calc.js:60). Mon/Wed/Fri program + multi-day water fasts mean honest rest/fasting days render as "Missed" for six months — trains him to lie-tap or distrust the grid. Fix: fast-aware and rest-aware day rules. → /impeccable shape
5. **[P2] Fasting stage transitions are anti-celebrations.** Crossing into ketosis snaps the ring from full to empty; the stage name is 11.5px muted text; ring never changes color. The coach voice (principle 4) doesn't exist anywhere. Fix: per-stage ring color, promoted stage label, one-line stage-reached message = natural home of the corner man. → /impeccable delight

## Persona Red Flags

**Casey (distracted mobile user, primary):** daily actions outside first screen and thumb zone; sub-44px targets; iOS focus-zoom every weigh-in; two CDN scripts block first render even at the signed-out gate.

**Sam (low-vision proxy for mid-40s eyes):** contrast passes AA (~8.5:1) but size violates the product's own 14px floor everywhere it matters — 11.5px carries the fasting stage, habit status, KPI labels, legends; color-only meaning on 6px calendar dots; no aria-live on errors/timer; weak focus indicator (outline:none, 1px border swap).

**Riley (stress tester):** habit double-tap races stale state; accidental End fast unrecoverable; two tabs desync activeFast; milestone table pinned to 1.5 lb/wk while the pace radio changes "Est. goal" — two disagreeing projections on one page.

**Micheal, 6:15 AM, pre-coffee (project persona):** first word is a sign-in flash, second is "258.4", third is a Sign out button. Mid-fast payoff (stage name) requires squinting. When the day counts, nothing says so. Because he fasted, the grid calls today "Missed."

## Minor Observations

- Nav padding differs across pages (pills shift horizontally page-to-page)
- .pill = static tag on Meal/Workout pages but pill-shape = button on tracker
- Native confirm()/alert() break the visual system
- Inline styles everywhere against a defined class system
- Coach line (principle 4) absent entirely
- cd-cell titles show date only, not status
- Fire emoji used twice with different meanings
- Table body 13.4px, under the 14px floor
- Progress bars animate width (detector); ring already does stroke-dashoffset correctly

## Questions to Consider

1. What does "a day that counts" mean during a 72-hour water fast? The two flagship features currently score against each other.
2. Should phone and desktop be the same page? Phone is a cockpit (timer, one number, two taps); desktop is trend review.
3. What if the fasting ring were the only circle? The donut is derived from weight by formula — it adds no information the KPI row doesn't.
